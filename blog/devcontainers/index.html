Dev Containers: Consistent Dev Environments for Java and Next.js

Stop the 'works on my machine' problem. Dev Containers let you define your entire development environment in code — here's how to set them up for Java and Next.js TypeScript projects.

Nikhil Shinde·March 12, 2026

The Problem Dev Containers Solve

You clone a repo, run the project, and it blows up. Wrong Node version. Missing JDK. A native dependency that only works on Linux. You spend two hours debugging your environment instead of the actual code.

Dev Containers fix this. You define your entire development environment — runtime, tools, extensions, environment variables — in a .devcontainer/devcontainer.json file that lives in the repo. Anyone who opens the project in VS Code (or GitHub Codespaces) gets the exact same environment, automatically.

How It Works

Under the hood, Dev Containers use Docker. When you open a project in VS Code with the Dev Containers extension, VS Code builds a container from your config, mounts your code into it, and attaches to it. You're coding inside Docker, but it feels like a normal editor.

Your files live on your machine. The runtime lives in the container. Best of both worlds.

Setting Up for a Java Project

Here's a .devcontainer/devcontainer.json for a Spring Boot project using Java 21 and Maven:

{
  "name": "Java Spring Boot",
  "image": "mcr.microsoft.com/devcontainers/java:21",
  "features": {
    "ghcr.io/devcontainers/features/java:1": {
      "version": "21",
      "installMaven": true,
      "installGradle": false
    },
    "ghcr.io/devcontainers/features/docker-in-docker:2": {}
  },
  "customizations": {
    "vscode": {
      "extensions": [
        "vscjava.vscode-java-pack",
        "vmware.vscode-spring-boot",
        "vscjava.vscode-spring-initializr",
        "redhat.vscode-xml"
      ],
      "settings": {
        "java.compile.nullAnalysis.mode": "automatic",
        "editor.formatOnSave": true
      }
    }
  },
  "forwardPorts": [8080],
  "postCreateCommand": "mvn dependency:resolve",
  "remoteUser": "vscode"
}

A few things worth noting:

  • image pulls a pre-built Microsoft devcontainer image with Java already installed. No Dockerfile needed for most cases.
  • features are composable add-ons — here we're adding Maven and Docker-in-Docker (useful if your Spring app talks to other containers).
  • forwardPorts automatically exposes port 8080 so you can hit localhost:8080 from your browser as normal.
  • postCreateCommand runs once after the container is created — here it pre-fetches Maven dependencies so your first build isn't slow.

Using a Custom Dockerfile Instead

If you need more control — say you need a specific version of a library or want to install extra tools — swap image for a build:

{
  "name": "Java Spring Boot",
  "build": {
    "dockerfile": "Dockerfile",
    "context": ".."
  }
}

And your Dockerfile:

FROM mcr.microsoft.com/devcontainers/java:21

# Install extra tooling
RUN apt-get update && apt-get install -y \
    httpie \
    jq \
    && rm -rf /var/lib/apt/lists/*

# Install a specific Maven version
ARG MAVEN_VERSION=3.9.6
RUN mvn --version || true

Setting Up for Next.js TypeScript

Here's a .devcontainer/devcontainer.json for a Next.js TypeScript project:

{
  "name": "Next.js TypeScript",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:22",
  "features": {
    "ghcr.io/devcontainers/features/node:1": {
      "version": "22",
      "nodeGypDependencies": true
    }
  },
  "customizations": {
    "vscode": {
      "extensions": [
        "bradlc.vscode-tailwindcss",
        "esbenp.prettier-vscode",
        "dbaeumer.vscode-eslint",
        "ms-vscode.vscode-typescript-next",
        "prisma.prisma"
      ],
      "settings": {
        "editor.defaultFormatter": "esbenp.prettier-vscode",
        "editor.formatOnSave": true,
        "typescript.tsdk": "node_modules/typescript/lib"
      }
    }
  },
  "forwardPorts": [3000],
  "postCreateCommand": "npm install",
  "remoteUser": "node"
}

This gives every developer on the team the same Node version, the same VS Code extensions (Tailwind IntelliSense, ESLint, Prettier), and runs npm install automatically on first open.

No more "my ESLint isn't running" because they have an old extension version.

Running Multiple Services with Docker Compose

Real projects usually need more than one container — your app, a database, maybe Redis. You can use Docker Compose with Dev Containers:

.devcontainer/
  devcontainer.json
  docker-compose.yml

docker-compose.yml:

version: "3.8"
services:
  app:
    build:
      context: ..
      dockerfile: .devcontainer/Dockerfile
    volumes:
      - ..:/workspace:cached
    command: sleep infinity
    environment:
      - DATABASE_URL=postgresql://postgres:postgres@db:5432/mydb

  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: mydb
    volumes:
      - postgres-data:/var/lib/postgresql/data

volumes:
  postgres-data:

devcontainer.json pointing at the compose file:

{
  "name": "App + Postgres",
  "dockerComposeFile": "docker-compose.yml",
  "service": "app",
  "workspaceFolder": "/workspace",
  "forwardPorts": [3000, 5432],
  "postCreateCommand": "npm install"
}

VS Code attaches to the app service, but the database is running alongside it. Your app can connect to db:5432 just like it would in production.

Useful Lifecycle Hooks

Dev Containers give you a few hooks to automate setup:

HookWhen it runs
initializeCommandOn the host, before the container starts
onCreateCommandInside the container, once, on first create
postCreateCommandAfter onCreateCommand, good for installs
postStartCommandEvery time the container starts
postAttachCommandEvery time VS Code attaches

Example — running DB migrations after create and starting the dev server on every attach:

{
  "postCreateCommand": "npm install && npx prisma migrate dev",
  "postAttachCommand": "npm run dev"
}

Tips

Use pre-built images when possible. The Microsoft devcontainer images (mcr.microsoft.com/devcontainers/*) are well-maintained and fast to pull. Only write a custom Dockerfile when you actually need something they don't provide.

Pin your image versions. Using java:21 is fine for a team project, but for a shared CI environment consider pinning to a digest to guarantee reproducibility.

Check in the .devcontainer folder. The whole point is that everyone uses the same environment. Commit it.

GitHub Codespaces uses the same config. If you ever want to spin up your project in a browser with zero local setup, it just works.

Wrapping Up

Dev Containers are one of those things that feel like overhead until you're onboarding someone new and they're up and running in five minutes instead of half a day. Define the environment once, check it in, forget about it.

If you're on a team project in Java or Next.js, it's worth the 20 minutes to set one up.
